# 180일 초기 성장 데이터셋 기준선

## 목적과 범위

이 문서는 서비스별 자원 사용량을 같은 데이터 크기에서 비교하기 위한 가상 사업 조건을 정의한다. 먼저 하루에 어떤 일이 몇 번 일어나는지 정하고, 그 값을 180일 동안 누적해 예상 행 수를 계산한다. 큰 숫자를 먼저 정한 뒤 원인을 끼워 맞추지 않는다.

이번 idle 벤치마크는 스키마 마이그레이션만 적용한 빈 데이터베이스를 사용한다. 이 문서는 뒤이어 만들 API 부하 테스트와 bulk seeder의 공통 기준이며, 이번 작업에서 180일치 데이터를 실제로 적재하지 않는다.

## 사업 가정

### 사용자와 서비스 기간

- 서비스 시작 시점 사용자: 10,000명
- 하루 신규 가입자: 500명
- 서비스 기간: 180일
- 180일 뒤 사용자: `10,000 + (500 x 180) = 100,000명`

탈퇴와 휴면은 첫 기준선에서 제외한다. 따라서 가입 이력과 현재 사용자를 분리하지 않고 100,000명을 최대 사용자 규모로 본다.

### 드롭과 상품

여기서 드롭(drop)은 정해진 시점에 상품을 공개하고 판매하는 묶음이다.

- 하루 신규 드롭: 6개
- 드롭당 상품: 12개
- 180일 드롭: `6 x 180 = 1,080개`
- 180일 상품: `1,080 x 12 = 12,960개`

현재 Catalog 모델에서 상품은 반드시 하나의 드롭에 속한다. 따라서 같은 상품을 여러 드롭이 공유하는 다대다 관계를 가정하지 않는다.

### 조회, 찜, 주문

모든 드롭의 인기가 같지 않으므로 1,080개 드롭을 세 등급으로 나눈다.

| 등급 | 비율 | 드롭 수 | 드롭당 조회자 | 구매 전환율 | 구매자당 주문 |
| --- | ---: | ---: | ---: | ---: | ---: |
| 일반 | 70% | 756 | 700 | 5% | 1.2 |
| 기획 | 25% | 270 | 1,500 | 10% | 1.2 |
| 행사 | 5% | 54 | 3,000 | 18% | 1.2 |

조회는 `756 x 700 + 270 x 1,500 + 54 x 3,000 = 1,096,200건`이다. 하루 평균으로는 `1,096,200 / 180 = 6,090건`이다.

찜은 일반 드롭 조회자의 8%, 기획 드롭의 12%, 행사 드롭의 18%가 한 번 남긴다고 가정한다.

- 일반 찜: `756 x 700 x 8% = 42,336건`
- 기획 찜: `270 x 1,500 x 12% = 48,600건`
- 행사 찜: `54 x 3,000 x 18% = 29,160건`
- 합계: `120,096건`, 하루 평균 `667.2건`

주문은 구매 전환 후 구매자 한 명이 평균 1.2개 주문을 만든다고 본다.

- 일반 주문: `756 x 700 x 5% x 1.2 = 31,752건`
- 기획 주문: `270 x 1,500 x 10% x 1.2 = 48,600건`
- 행사 주문: `54 x 3,000 x 18% x 1.2 = 34,992건`
- 합계: `115,344건`, 하루 평균 `640.8건`

현재 Order 계약은 주문 한 건에 상품 한 개를 담는다. 장바구니 한 번에 여러 상품을 결제하는 모델이 아니므로 상품 수량과 주문 행 수를 합쳐 계산하지 않는다.

### 결제

주문마다 terminal payment, 즉 승인 또는 최종 실패로 끝난 결제 한 건을 둔다. 승인율은 약 92%로 가정한다.

- 결제 합계: 주문과 같은 `115,344건`
- 승인 결제: `106,116건`
- 최종 실패 결제: `115,344 - 106,116 = 9,228건`

재시도 요청은 idempotency와 충돌 검증을 시험하는 API 사건으로는 만들 수 있지만, 현재 Payment는 주문당 terminal payment 충돌을 검사하므로 별도 terminal payment 행을 계속 추가하지 않는다.

### 쿠폰

- 정기 캠페인: 하루 2개씩 `2 x 180 = 360개`
- 계절·행사 캠페인: 40개
- 캠페인 합계: 400개
- 일반 발급: 하루 1,600건씩 `1,600 x 180 = 288,000건`
- 행사 추가 발급: 12,000건
- 사용자 쿠폰 합계: 300,000건
- 사용 완료: 주문의 약 30%인 `34,603건`

쿠폰 발급, 예약, 사용, 만료는 상태 변경 이력을 남긴다. bulk seeder를 만들 때에는 현재 Coupon 스키마의 캠페인, 사용자 쿠폰, redemption, command/outbox 테이블 관계를 지켜야 한다.

### 알림

주문 한 건에서 평균 네 번의 알림 요청이 생긴다고 가정한다. 예를 들면 주문 접수, 결제 결과, 주문 확정 또는 실패, 후속 안내다.

- 알림 요청 이벤트: `115,344 x 4 = 461,376건`
- Kafka 소비가 모두 성공한 경우 알림 행도 최대 461,376건
- 하루 평균: `461,376 / 180 = 2,563.2건`

Notification은 업무 API로 알림을 직접 생성하지 않고 Kafka 이벤트를 소비해 생성한다. 따라서 seeder는 알림 테이블만 채우는 방식과 실제 이벤트 소비 방식을 구분해야 한다.

## 서비스별 엔티티 매핑

| 서비스 | 데이터셋 항목 | 대표 엔티티와 관계 |
| --- | --- | --- |
| Auth | 가입, 로그인 자격, 세션, 감사·전달 작업 | identity/credential, session, challenge, audit·delivery outbox. 사용자와 인증 데이터는 같은 식별자를 사용하되 소유권은 나눈다. |
| User | 사용자, 약관 동의, 프로필 | user, agreement acceptance, profile. 초기 10,000명과 일 신규 500명을 누적한다. |
| Catalog | 드롭, 상품, 재고 | drop 1개가 product 12개를 소유한다. 현재 상품 등록 API가 없으므로 향후 seeder는 DB fixture 또는 별도 관리 계약이 필요하다. |
| Interest | 조회, 찜, 랭킹 입력 | view event와 favorite를 사용자·드롭·상품에 연결하고 중복 정책을 분리한다. |
| Order | 주문 | order 한 건은 사용자 한 명, 드롭 한 개, 상품 한 개를 참조한다. outbox와 만료 작업을 함께 고려한다. |
| Payment | 승인·최종 실패 결제, 환불 작업 | order당 terminal payment 한 건이며 중복 terminal 생성 충돌을 보존한다. |
| Notification | 알림 요청 이벤트, 알림 | Kafka의 notification.requested 같은 이벤트를 소비해 사용자 알림을 만든다. |
| Coupon | 캠페인, 사용자 쿠폰, 사용 예약·완료 | campaign에서 user coupon을 발급하고 order/payment 처리와 redemption을 연결한다. |

## 현재 코드 제약과 해석 주의사항

- Catalog에는 상품 등록 API가 없다. 상품은 하나의 drop에 속하므로 데이터 생성 도구가 현재 DB 계약을 직접 지킬 방법을 별도로 설계해야 한다.
- Order는 주문당 상품 하나다. 여러 상품을 한 주문에 넣는 장바구니 가정은 사용하지 않는다.
- Payment는 주문당 terminal payment 충돌을 검사한다. 결제 재시도를 terminal 행의 무제한 증가로 계산하지 않는다.
- Notification은 Kafka 이벤트로 생성된다. DB 직접 적재는 소비자 비용을 재현하지 못한다.
- Dropmong Web의 checkout과 주문 결과는 현재 개발 mock이다. 웹 fixture 성공을 실제 Order·Payment 연동 성공으로 해석하지 않는다.
- idle 결과는 API 처리량을 뜻하지 않는다. 데이터가 비어 있는 idle과 180일 데이터가 적재된 idle도 서로 다른 실험이다.

## 이번 idle 실행의 데이터 상태

idle runner는 각 서비스를 고유한 Compose project와 새 볼륨에서 시작하고 마이그레이션만 적용한다. 사용자·드롭·상품·주문 같은 업무 데이터는 넣지 않는다. 이 사실은 각 run의 `execution.json`과 서비스 JSON에 `schema_only`, `business_rows_seeded: false`로 기록한다.
